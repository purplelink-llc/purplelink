## Option 1

When I started building the PDF to Structured Data tool, the first question was whether to call a third-party extraction API or run something locally. Most services that do PDF-to-Markdown well charge per page, require API keys, and your document goes to their infrastructure.

I went a different direction. OpenDataLoader-PDF is an Apache-2.0 library from Hancom. It is deterministic, runs locally, makes no model calls, and has no per-page billing. It shells out to a JRE, parses the PDF's text layer, and writes reading-order Markdown plus a structured JSON tree. I run it inside a dedicated Modal function: an isolated container that spins up when the endpoint gets a request and disappears after.

The honest limitation: it only works on PDFs that already have a text layer. Scanned papers need OCR, and that is not in this version. I wanted something that works reliably on most modern academic papers before tackling the harder scanned-paper path.

The tool is free at purplelink.llc/tools/pdf-structure/. Tables come out as Markdown tables, math comes out as LaTeX, and figure captions are captured. Nothing is stored. The full attribution for the open-source components is at purplelink.llc/tools/pdf-structure/ATTRIBUTION.md.

If you work with academic PDFs and have a chance to test the extraction quality on your papers, I would be interested to hear how it holds up.

## Option 2

Most online tools that process uploaded files have a privacy policy that says your data "may be used to improve our service" or is "retained for N days." That is not necessarily dishonest, but it means the file left your machine and is sitting somewhere.

I built the PDF to Structured Data tool differently because the people most likely to use it are working with drafts that have not been published yet. Submitting an unpublished manuscript to a third-party API to convert it to Markdown is a real risk, and most researchers do not stop to check whether that is happening.

The way the tool works: you upload the PDF, it goes to an ephemeral container running our extraction function, the result comes back over HTTPS, and the container is gone. Nothing is written to a database. Nothing is written to logs. No third-party service sees the file. The extraction engine, OpenDataLoader (Apache-2.0), runs entirely on Purplelink's own infrastructure.

This is an architecture decision, not a marketing statement. The attribution file at purplelink.llc/tools/pdf-structure/ATTRIBUTION.md lists exactly which open-source components are running. The tool is free and the output is reading-order Markdown plus structured JSON.

If you are careful about where your unpublished work goes, I would be curious whether this kind of transparency actually matters to you in practice.

## Option 3

A PDF is designed to look right on screen, not to be processed by software. Internally it is a flat list of positioned text spans with no concept of reading order across columns, no way to know which text belongs to a table, and nothing to distinguish a figure caption from body text. When you copy-paste from a two-column paper into a text editor and the paragraphs come out scrambled, that is the same underlying problem.

I shipped a free tool this week that addresses this for academic papers. You upload a PDF and get back two things: reading-order Markdown with tables formatted as Markdown tables and math written as LaTeX, and a structured JSON tree where every block has a type, page number, and bounding box. The JSON is designed for retrieval pipelines: you can filter by section, chunk by block type, and know which page a passage came from.

The extraction runs on OpenDataLoader-PDF (Apache-2.0), a deterministic library with no model calls. The current version handles PDFs that already have a text layer. Scanned papers need OCR and that is not in this version.

The tool is free at purplelink.llc/tools/pdf-structure/ and your file is processed in memory and discarded immediately.

If you have been dealing with the PDF ingestion step in a research workflow, I would like to hear what the output was missing for your use case.

## Selected

Option 3 -- it explains a concrete problem researchers will recognize from their own experience (the copy-paste scramble), makes the tool's value specific and immediate, and the limitation is stated plainly before the call to action.
